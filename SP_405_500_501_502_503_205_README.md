## SP-405: Enhanced PayoutAgent

**Location:** `shieldpoint_agents/src/shieldpoint_agents/v2/payout/`

The PayoutAgent has been enhanced from a simple idempotency-checking stub to
a full payment processing pipeline with seven steps:

1. **Duplicate payment detection** — queries the payment ledger by
   idempotency key (`payout-{claim_id}`) before initiating any payment.
2. **Bank details verification** — calls `BankVerificationService` to
   validate the payee's bank account and routing number.
3. **ACH payment initiation** — calls `ACHProvider.initiate_payment()`
   to initiate the electronic funds transfer. Returns an ACH reference
   number and expected settlement date.
4. **Payment record creation** — inserts an immutable `PaymentRecord`
   into the ledger with the full breakdown (gross, deductible, co-pay, net).
5. **PDF receipt generation** — `ReceiptGenerator` produces a
   professional PDF receipt with the payment breakdown, ACH details,
   ZKP proof references, and audit trail.
6. **Claimant notification** — `NotificationService` sends an email to
   the claimant with the payment confirmation and receipt attachment.
7. **Audit record assembly** — `AuditRecordAssembler` assembles the
   complete audit record (all agent traces + ZKP proofs + payment
   breakdown) with a SHA-256 content hash for integrity.

### Files
- `payout/ach_provider.py` — `StubACHProvider`, `BankVerificationService`
- `payout/ledger.py` — `InMemoryPaymentLedger`, `PaymentRecord`, `check_duplicate`, `compute_payment_breakdown`
- `payout/receipt_generator.py` — `ReceiptGenerator` (PDF via ReportLab, text fallback)
- `payout/notification.py` — `StubNotificationService`, `SMTPNotificationService`
- `payout/audit_record.py` — `AuditRecordAssembler`, `AuditRecord`, `AgentTrace`, `ZKPProofRef`

### Guard Condition
The `APPROVED → PAID_OUT` guard (in `state_machine.py`) checks:
- `payment_authorized == True`
- `bank_details_verified == True`

### Tests
`tests/v2/test_payout_agent.py` — 33 tests covering ACH, ledger, receipt,
notification, audit record, and end-to-end PayoutAgent integration.

## SP-205: LoRA Fine-Tuning Pipeline

**Location:** `training/`

Full QLoRA (Quantized LoRA) pipeline for fine-tuning Qwen3.6 35B A3B on
ShieldPoint's 3,500 historical claims.

### Configuration
- **Base model**: Qwen3.6-35B-A3B (4-bit NF4 quantized via BitsAndBytes)
- **LoRA rank**: 16
- **LoRA alpha**: 32
- **Target modules**: `q_proj`, `v_proj` (query and value projections)
- **Optimizer**: paged AdamW 8-bit with cosine LR schedule
- **Epochs**: 3, Batch size: 4 (×4 grad accum = 16 effective)

### Scripts
- `prepare_training_data.py` — Converts claims JSONL → Alpaca instruction format
- `lora_train.py` — QLoRA training script (requires torch, transformers, peft, trl)
- `evaluate_adapter.py` — Evaluates adapter on validation set, compares to base
- `mmlu_benchmark.py` — MMLU subset for general reasoning retention check
- `package_adapter.py` — Packages adapter for LM Studio loading
- `weekly_eval_pipeline.py` — Weekly evaluation + promotion decision

### Acceptance Criteria
- ✅ LoRA rank=16, alpha=32, target_modules=[q_proj, v_proj]
- ✅ QLoRA 4-bit quantization (NF4)
- ✅ Training data in instruction-following format
- ✅ Evaluation computes classification accuracy + MAE
- ✅ MMLU retention check (within 2pp threshold)
- ✅ Adapter packaging with config + metrics + README
- ✅ Weekly pipeline with promotion decision logic

### Tests
`tests/v2/test_lora_training.py` — 17 tests covering data prep, config,
evaluation, MMLU, packaging, and weekly pipeline.

## SP-500: Cross-Party ZKP Fraud Detection Network

**Location:** `zkp_circuit/fraud_detection/`

Enables ShieldPoint to detect duplicate claims filed with other insurers
WITHOUT exposing any sensitive data. Three components:

### 1. Commitment Generation Service (`commitment.py`)
- Creates Poseidon hashes of `claimant_id || incident_hash || salt`
- Only the commitment (a single field element) is shared — never raw data
- Supports `poseidon_py` and falls back to SHA-256 simulation for testing

### 2. Shared Merkle Tree (`merkle_tree.py`)
- Sorted Merkle tree of commitments (ascending order)
- Supports incremental insertion — O(depth) per operation
- Generates both membership and non-membership proofs
- Handles sentinel cases (tree boundaries — left/right edges)
- Configurable depth (default 20 = 2^20 = 1M leaves capacity)

### 3. Fraud Detection Client (`client.py`)
- High-level client orchestrating commitment generation + coordination layer query
- `check_claim_uniqueness()` — main entry point called by ClassifierAgent
- Auto-submits commitment to shared tree after successful uniqueness check
- Defence-in-depth: re-verifies coordination layer proofs locally

### Tests
`tests/v2/test_fraud_detection.py` — 38 tests covering commitments,
Merkle tree operations, prover, client, and multi-insurer integration.

## SP-501: ZKP Non-Membership Proof Circuit

**Location:** `zkp_circuit/circuits/non_membership.circom`

The Circom circuit proving a commitment is NOT in the shared Merkle tree.

### Circuit Design
- **Public inputs**: `merkleRoot`, `newCommitment`
- **Private inputs**: `leftNeighbor`, `rightNeighbor`, Merkle paths, sentinel flags
- **Non-membership logic**: proves `leftNeighbor < newCommitment < rightNeighbor`
  (strict bracketing) + validates both neighbors' Merkle paths to the root
- **Sentinel handling**: left/right boundary cases (commitment smaller/larger
  than all leaves) handled via conditional constraints
- **Tree depth**: 20 (supports 2^20 = 1,048,576 leaves)

### Constraint Budget
- **Target**: ≤ 80,000 constraints
- **Estimated**: ~10,600 constraints (2× Merkle path verification × 20 levels
  × ~250 Poseidon constraints + comparators)

### Python Wrapper (`non_membership_prover.py`)
- `ProverMode.AUTO` — tries Groth16, falls back to stub
- `ProverMode.GROTH16` — real Circom + snarkjs via subprocess
- `ProverMode.STUB` — deterministic Python stub (for CI/development)
- Generates + verifies proofs; measures latency

### Makefile Targets
```bash
make compile-fraud          # Compile non_membership.circom
make fraud-trusted-setup    # Groth16 trusted setup (8 contributors)
make test-fraud             # Run fraud detection tests
```

### Tests
`tests/v2/test_non_membership_circuit.py` — 18 tests covering circuit
structure, constraint budget, Poseidon usage, test vectors, and Makefile targets.

## SP-502: Inter-Insurer Coordination Layer API

**Location:** `coordination_layer/`

FastAPI service serving as the communication backbone for the fraud
detection network.

### Endpoints
| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/v1/root` | Current Merkle tree root |
| POST | `/api/v1/commitments` | Submit a new commitment |
| GET | `/api/v1/proofs/membership/{c}` | Membership proof |
| GET | `/api/v1/proofs/non-membership/{c}` | Non-membership proof |
| GET | `/api/v1/stats` | Tree statistics |
| WS | `/api/v1/notifications` | WebSocket for tree update notifications |
| GET | `/health` | Health check |

### Security
- **Mutual TLS**: all endpoints require a client certificate (config via
  `--ssl-certfile`, `--ssl-keyfile`, `--ssl-ca-certs`)
- **Data minimisation**: stores ONLY Poseidon hash values + Merkle metadata
- **Audit logging**: every submission logs the insurer's cert CN + timestamp

### Persistence
- **PostgreSQL** (production): `DATABASE_URL` env var
- **SQLite** (development/tests): auto-fallback

### Running
```bash
# Development (SQLite, no mTLS)
uvicorn coordination_layer.api:app --reload --port 8000

# Production (mTLS, PostgreSQL)
uvicorn coordination_layer.api:app \
    --host 0.0.0.0 --port 8443 \
    --ssl-certfile /etc/shieldpoint/server.crt \
    --ssl-keyfile /etc/shieldpoint/server.key \
    --ssl-ca-certs /etc/shieldpoint/ca.crt
```

### Tests
`tests/v2/test_coordination_layer.py` — 15 tests covering all endpoints,
proof generation, notification subscriptions, and the 2-second proof
generation performance requirement.

## SP-503: Fraud Detection Integration into Agent Workflow

**Location:** `shieldpoint_agents/src/shieldpoint_agents/v2/agents.py`

Integrates the cross-party fraud detection network into the state machine.

### ClassifierAgent Enhancement
- Accepts an optional `fraud_detection_client` parameter
- During the `CLASSIFYING` state, queries the fraud detection network
- Generates a commitment for the claim and checks it against the shared tree
- If a duplicate is detected (`is_unique=False`):
  - Sets `fraud_flag=True` in the context
  - Sets `ambiguous=True` to force routing to ESCALATING
  - Records the duplicate insurer and ZKP proof failure details

### EscalationAgent Enhancement
- When `fraud_flag=True`, the case summary includes a `fraud_investigation` section:
  - Fraud flag + reason
  - Duplicate insurer + commitment value
  - ZKP proof failure details
  - Investigation steps (5 recommended actions)
  - Recommended action: `deny` if duplicate confirmed, `reclassify` if coincidental
- `_infer_escalation_reason()` checks fraud flag first (highest priority)
- `_recommend_next_steps()` includes cross-party fraud investigation steps

### State Machine Flow
```
CLASSIFYING
    ↓ (fraud detection check)
    ├── is_unique=True → continue to ZKP_COMPLIANCE_PROOF
    └── is_unique=False (duplicate) → fraud_flag=True
        → ambiguous=True → compliance guard fails
        → ESCALATING (with fraud_investigation case)
            ├── adjuster denies (duplicate confirmed) → ESCALATING terminal
            └── adjuster reclassifies (coincidental) → APPROVED → PAID_OUT
```

### Tests
`tests/v2/test_fraud_integration.py` — 13 tests covering ClassifierAgent
fraud detection, EscalationAgent fraud investigation, adjuster decisions,
and end-to-end duplicate claim detection flow.
